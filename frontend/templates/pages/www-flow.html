Component Deep Dives

1. DNS Resolver

What: Translates domain names (blinkit.com) to IP addresses
Why: Human-readable domains, traffic steering, failover capabilities
If Missing: Users must type raw IPs; no domain-based routing; impossible to rotate infrastructure
Alternatives: Route53, Cloudflare DNS, NS1, Google Cloud DNS
Best Now: Cloudflare + Route53 combo for speed + infrastructure-as-code
Future: More geo/latency-aware routing, DNSSEC everywhere, tighter service discovery
  • Stub Resolver: On your device - first stop for DNS queries
  • Recursive Resolver: ISP or public (1.1.1.1, 8.8.8.8) - does the heavy lifting
  • Root Servers: 13 clusters worldwide - point to TLD servers
  • TLD Servers: Handle .com, .org, .io etc.
  • Authoritative NS: Your domain's actual DNS records
Real Example: Swiggy

Swiggy uses Route53 with latency-based routing. Users in Mumbai get directed to AWS Mumbai, users in Bangalore to AWS Bangalore. DNS TTL is 60 seconds for quick failover during outages.

2. CDN / Edge

What: Edge network caching static assets and sometimes API responses near users
Why: Lower latency, offload origin servers, DDoS protection
If Missing: Higher TTFB, origin server overload during traffic spikes
Alternatives: Cloudflare, Akamai, Fastly, CloudFront
Best Now: Cloudflare/Fastly for dynamic sites; Akamai for very large enterprises
Future: More edge compute, AI-driven cache policies, HTTP/3+ protocol upgrades
Cloudflare CloudFront Fastly Akamai
Real Example: Blinkit

Blinkit uses Cloudflare CDN with 200+ PoPs globally. Product images are cached at edge with 24-hour TTL. Their cache hit ratio is 85%+, reducing origin load by 6x during Diwali sales.

3. Load Balancer

What: Distributes incoming traffic across multiple backend instances
Why: Horizontal scalability, zero-downtime deploys, health checks
If Missing: Single instance bottleneck, no rolling upgrades, single point of failure
Alternatives: AWS ALB/NLB, GCP LB, NGINX, HAProxy, Envoy
Best Now: Managed LBs (ALB) externally + NGINX/Envoy inside Kubernetes
Future: Smarter routing (per-user, cost-aware), more L7 security built-in
  • Round Robin: Rotate through servers sequentially
  • Least Connections: Send to server with fewest active connections
  • Weighted: Distribute based on server capacity
  • IP Hash: Same client IP always goes to same server (sticky sessions)
  • Least Response Time: Send to fastest responding server
AWS ALB NGINX HAProxy Envoy

4. API Gateway

What: Single entry point for APIs: routing, auth, rate limits, observability
Why: Avoid duplicating cross-cutting logic across microservices
If Missing: Every service reimplements auth, throttling, logging, CORS
Alternatives: Kong, Apigee, AWS API Gateway, Traefik, Istio Ingress
Best Now: Kong/Envoy-based gateways + service mesh for large microservices
Future: Policy-as-code, AI-assisted API governance, unified API catalogs
Real Example: Grofers/Blinkit

Blinkit uses Kong API Gateway handling 50K+ requests/second. Their gateway handles JWT validation, rate limiting (100 req/min for anonymous, 1000 for logged-in), request transformation, and metrics collection.

Kong AWS API Gateway Apigee Traefik

5. Backend Microservices

What: Individual services handling specific domains (Auth, Catalog, Orders, etc.)
Why: Independent scaling, team ownership, technology flexibility, resilience
If Missing: Monolith grows too large; deployment and scaling become painful
Alternatives: Monolith, modular monolith, microservices, serverless
Best Now: Microservices/modular monolith with clear bounded contexts
Future: More event-driven designs (Kafka), function-level scaling (serverless)
FastAPI Go Gin Spring Boot Node.js

6. Database

What: Systems of record - user profiles, catalog, orders, transactions
Why: Consistent, durable storage with ACID guarantees
If Missing: No persistence, can't reconcile orders/payments, data loss
Alternatives: PostgreSQL, MySQL, MongoDB, Cassandra, DynamoDB
Best Now: PostgreSQL for transactional data, MongoDB for flexible documents
Future: Serverless DBs, better multi-region consistency, HTAP systems
  • Primary-Replica: One write node, multiple read replicas
  • Sharding: Horizontal partition data by key (user_id, region)
  • Read/Write Splitting: Route reads to replicas, writes to primary
  • Connection Pooling: PgBouncer, ProxySQL to manage connections
PostgreSQL MySQL MongoDB DynamoDB

7. Cache (Redis)

What: In-memory store for hot data (catalog, sessions, rate limit counters)
Why: Reduce DB load, sub-millisecond latency for frequent reads
If Missing: Database hot spots, higher latency for high-traffic flows
Alternatives: Redis, Memcached, in-process caches (Caffeine, Guava)
Best Now: Redis for most web apps (data structures, persistence options)
Future: Redis-compatible high-performance engines (Dragonfly), serverless cache
  • Cache-Aside: App checks cache, if miss reads DB and populates cache
  • Write-Through: Writes go to cache AND DB synchronously
  • Write-Behind: Writes go to cache, async write to DB later
  • Read-Through: Cache fetches from DB on miss automatically
Redis Memcached ElastiCache

8. Message Queue

What: Durable async pipeline for events (order placed, payment processed)
Why: Decouple producers/consumers, handle traffic spikes, ensure delivery
If Missing: Synchronous flows everywhere, user-facing latency spikes, lost events
Alternatives: RabbitMQ, Kafka, SQS, Redis Streams
Best Now: Kafka for high-throughput event streams, RabbitMQ/SQS for task queues
Future: Unified event mesh, more exactly-once semantics, better observability
Real Example: Swiggy Order Flow

When you place an order, Swiggy publishes to Kafka. Restaurant service, delivery service, notification service all consume independently. If notification service is slow, orders still process. Kafka handles 100K+ events/second during peak hours.

Kafka RabbitMQ SQS Redis Streams

9. Background Workers

What: Processes that consume queue jobs: send emails, process payments, update inventory
Why: Keep request/response fast; handle retries and backoffs gracefully
If Missing: You'd block user requests doing slow work, poor user experience
Alternatives: Celery, Sidekiq, custom consumers, Lambda triggered from queues
Best Now: Language-native workers (Celery for Python, Sidekiq for Ruby) + managed queues
Future: More serverless event handlers, autoscaling based on queue lag
Celery Sidekiq Bull Lambda

10. Observability

What: Metrics (Prometheus), Logs (ELK), Traces (Jaeger) - the three pillars
Why: Debug production issues, capacity planning, SLO tracking
If Missing: Blind in production, MTTR explodes, can't identify bottlenecks
Alternatives: Datadog, New Relic, OpenTelemetry stack, Grafana Cloud
Best Now: OpenTelemetry + managed observability (Datadog/Grafana Cloud)
Future: AI-assisted incident detection, automated remediation playbooks
Prometheus Grafana Jaeger ELK Stack Datadog
